# Architecture Overview

## 1. ภาพรวมของระบบ
ระบบแจ้งเหตุเพลิงไหม้เป็นเว็บแอปพลิเคชันต้นแบบที่ออกแบบมาเพื่อแสดงสถานะการตรวจจับควันและเปลวไฟ พร้อมระบบแจ้งเตือนและประวัติเหตุการณ์ โดยแบ่งออกเป็น 2 ส่วนหลักคือ

- Frontend: แสดงข้อมูลและโต้ตอบกับผู้ใช้ผ่านเว็บ UI
- Backend/API: จัดการข้อมูลสถานะและการแจ้งเตือนให้ frontend เรียกใช้

ระบบนี้เน้นความง่ายในการพัฒนาและต่อยอดในอนาคต โดยใช้ Next.js เป็นกรอบหลักและคงโครงสร้างให้แยกหน้าที่ชัดเจน เพื่อให้สามารถเพิ่มฟีเจอร์ เช่น การเชื่อมต่อเซ็นเซอร์จริง หรือฐานข้อมูลจริง ได้ในภายหลัง

---

## 2. โครงสร้างระบบ

### 2.1 ส่วนประกอบหลัก
- Client Layer
  - เว็บเบราว์เซอร์ของผู้ใช้
  - แสดงหน้า Dashboard, Alert Banner, Alert History

- Presentation Layer
  - หน้าแสดงสถานะระบบ เช่น หน้าแรกและส่วน UI หลัก
  - คอมโพเนนต์สำหรับแสดงข้อความเตือนและประวัติเหตุการณ์

- API Layer
  - Next.js Route Handlers สำหรับรับคำขอจาก frontend
  - จัดการข้อมูลสถานะล่าสุดและประวัติการแจ้งเตือน

- Service/Logic Layer
  - เก็บ logic สำหรับประมวลผลข้อมูลเหตุการณ์และเตรียมข้อมูลให้ UI ใช้งาน

- Data Layer
  - ในรุ่นต้นแบบอาจใช้ข้อมูลจำลองหรือข้อมูลในหน่วยความจำ
  - ในอนาคตสามารถเชื่อมต่อกับฐานข้อมูลหรือระบบภายนอกได้

### 2.2 โครงสร้างไฟล์ที่คาดหวัง
```text
app/
  page.tsx              # หน้า Dashboard หลัก
  layout.tsx            # Layout ของเว็บแอป
  api/
    status/route.ts     # API สำหรับสถานะปัจจุบัน
    alerts/route.ts     # API สำหรับการแจ้งเตือน
    history/route.ts    # API สำหรับประวัติเหตุการณ์
components/
  StatusDisplay.tsx     # แสดงสถานะการตรวจจับ
  AlertBanner.tsx       # แสดงข้อความเตือน
  AlertHistory.tsx      # แสดงประวัติเหตุการณ์
lib/ or services/
  api.ts                # ฟังก์ชันเรียก API
  utils.ts              # ฟังก์ชันช่วยทั่วไป
types/
  index.ts              # โครงสร้างข้อมูลประเภท
```

## 3. Data Flow

### 3.1 Flow การดูสถานะระบบ
1. ผู้ใช้เปิดหน้า Dashboard
2. Frontend ส่งคำขอไปที่ API สำหรับสถานะระบบ
3. API Layer ดึงข้อมูลสถานะล่าสุดจาก service หรือข้อมูลจำลอง
4. ข้อมูลถูกส่งกลับไปยัง UI
5. UI แสดงสถานะว่าเป็นปกติ/อันตราย/ไม่ทราบสถานะ

### 3.2 Flow การแจ้งเตือน
1. เมื่อมีเหตุการณ์ที่ต้องแจ้งเตือน ระบบสร้าง event ของ alert
2. Alert ถูกส่งผ่าน API Layer ไปยัง frontend
3. UI แสดง AlertBanner เพื่อให้ผู้ใช้รับรู้ทันที
4. ข้อมูล alert ถูกบันทึกไว้ในประวัติเหตุการณ์

### 3.3 Flow การดูประวัติ
1. ผู้ใช้เปิดส่วนประวัติการแจ้งเตือน
2. Frontend เรียก API history
3. API ส่งรายการเหตุการณ์ย้อนหลังกลับมา
4. UI แสดงข้อมูลพร้อมวันเวลาและระดับความรุนแรง

## 4. การตัดสินใจด้านดีไซน์หลัก

### 4.1 แยก Frontend และ Backend/API อย่างชัดเจน
เหตุผล: ทำให้โค้ดดูง่ายขึ้น เข้าใจง่าย และรองรับการพัฒนาแบบ modular

### 4.2 ใช้ Next.js App Router
เหตุผล: เหมาะกับการสร้างเว็บแอปที่มีหน้าเว็บและ API route อยู่ในโปรเจกต์เดียวกัน ลดความซับซ้อนในการ deploy

### 4.3 แยก Service Layer จาก UI Component
เหตุผล: ลดการผูกติดระหว่าง UI กับ business logic ทำให้แก้ไขหรือเปลี่ยนแปลงข้อมูลได้ง่ายขึ้น

### 4.4 ใช้ประเภทข้อมูล (TypeScript types) ชัดเจน
เหตุผล: ช่วยลดข้อผิดพลาดจากข้อมูลและทำให้การพัฒนาต่อยอดสะดวกขึ้น

### 4.5 เน้น UI ที่เข้าใจง่ายและตอบสนองเร็ว
เหตุผล: ระบบนี้เป็นต้นแบบสำหรับการแจ้งเหตุการณ์ จึงต้องให้ผู้ใช้รับรู้สถานะได้ทันทีและไม่สับสน

### 4.6 ออกแบบให้รองรับการขยายในอนาคต
เหตุผล: ในอนาคตสามารถเติม
- การเชื่อมต่อเซ็นเซอร์จริง
- ฐานข้อมูลแบบถาวร
- การแจ้งเตือนผ่านอีเมล/LINE/Telegram
- การจัดการผู้ดูแลและสิทธิ์การเข้าถึง

## 5. ข้อจำกัดของสถาปัตยกรรมปัจจุบัน
- ยังเป็นต้นแบบ จึงใช้ข้อมูลจำลองหรือข้อมูลพื้นฐานแทนการเชื่อมต่อกับอุปกรณ์จริง
- ฟังก์ชันการบันทึกข้อมูลยังขึ้นกับรูปแบบที่เลือกในอนาคต
- ความปลอดภัยและการยืนยันตัวตนยังเป็นระดับพื้นฐานเท่าที่จำเป็นสำหรับต้นแบบ

---

## 6. ข้อสรุป
สถาปัตยกรรมของระบบนี้ถูกออกแบบให้เป็นเว็บแอปพลิเคชันที่มีความเรียบง่าย แต่มีโครงสร้างที่ยืดหยุ่นและสามารถต่อยอดได้ โดยเน้นหลัก 3 เรื่องคือ

- แสดงสถานะระบบอย่างชัดเจน
- แจ้งเตือนเหตุการณ์ได้ทันที
- จัดเก็บและแสดงประวัติเหตุการณ์ได้อย่างเป็นระเบียบ
