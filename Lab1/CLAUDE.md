# CLAUDE.md

## Project Overview
โปรเจกต์นี้เป็นระบบต้นแบบสำหรับแจ้งเหตุเพลิงไหม้ ด้วย Next.js เป็นเฟรมเวิร์กหลัก โดยเน้นการสร้างเว็บแอปพลิเคชันที่แสดงสถานะการตรวจจับควันและเปลวไฟ ส่งการแจ้งเตือน และดูประวัติเหตุการณ์อย่างชัดเจน ระบบควรออกแบบให้แยกส่วน UI และ API เพื่อพัฒนาและต่อยอดได้ง่าย

## Stack ที่ใช้จริง
- Framework: Next.js
- Rendering: ใช้ App Router ของ Next.js เป็นหลัก
- UI: React Components + Tailwind CSS (ถ้ามี) หรือ CSS Module / Styled Component ตามสภาพแวดล้อม
- API: ใช้ Route Handlers ใน Next.js หรือแยก service layer สำหรับ logic
- Data: เก็บข้อมูลสถานะล่าสุด ประวัติการแจ้งเตือน และข้อมูลเหตุการณ์พื้นฐาน
- Deployment: เหมาะสำหรับ Vercel หรือเซิร์ฟเวอร์ที่รองรับ Node.js

## Project Structure ที่ควรยึด
- app/ : หน้าเว็บและ Route ของ Next.js
  - app/page.tsx : หน้าแรก
  - app/layout.tsx : layout หลัก
  - app/api/ : API routes ของ Next.js
- components/ : คอมโพเนนต์ UI ที่สามารถ reuse ได้
- lib/ หรือ services/ : logic การเชื่อมข้อมูลและธุรกิจของระบบ
- types/ : type definitions สำหรับข้อมูลของระบบ
- public/ : ไฟล์สาธารณะ เช่น รูปภาพหรือ icon

## Build / Development Commands
ใช้คำสั่งต่อไปนี้เป็นหลักเมื่อโปรเจกต์มี package.json อยู่แล้ว:

```bash
npm install
npm run dev
npm run build
npm run start
npm run lint
```

คำแนะนำเพิ่มเติม:
- ใช้ npm หรือ pnpm ตาม lockfile ที่มีอยู่ในโปรเจกต์
- ถ้าเพิ่ม dependency ใหม่ ให้ติดตั้งผ่าน package manager เดียวกันเสมอ
- หลังแก้ไขหน้าใหม่หรือ API ควรรัน build ก่อนส่งงาน

## Next.js Conventions ที่ควรปฏิบัติ
- ใช้ App Router แทน Pages Router หากเป็น Next.js เวอร์ชันใหม่
- แยกหน้า UI ออกจาก business logic
- สำหรับข้อมูลที่ต้องอัปเดตแบบไดนามิก ให้ใช้ Server Components หรือ Client Components อย่างเหมาะสม
- ถ้าใช้ client-side interactivity ให้ใส่ 'use client' เฉพาะจุดที่จำเป็น
- หลีกเลี่ยงการวาง logic หนักเกินไปใน component ที่แสดงผลโดยตรง
- ใช้ Route Handlers ใน app/api/ สำหรับ endpoint ที่จำเป็น
- ควรคงโครงสร้างไฟล์ให้อยู่ในรูปแบบที่อ่านง่ายและต่อยอดได้

## Coding Conventions
- ตั้งชื่อไฟล์และชื่อตัวแปรให้สื่อความหมายชัดเจน เช่น alertHistory, smokeStatus, fireAlert
- ใช้ชื่อตัวแปรและฟังก์ชันเป็นภาษาอังกฤษเป็นหลัก
- แยกโค้ดตามหน้าที่ เช่น components, app, lib, services, types
- ลดการซ้ำซ้อนของโค้ดและรักษาความเป็น modular
- เขียนคอมเมนต์หรือคำอธิบายเฉพาะจุดที่ซับซ้อนหรือไม่ชัดเจน
- จัดการ UI ให้ข้อความแจ้งเตือนเข้าใจง่ายและสั้นชัด
- สำหรับหน้าและคอมโพเนนต์ใหม่ ควรพิจารณา reusable และ maintainable ก่อน

## กฎสำคัญที่ต้องปฏิบัติ
- ต้องรักษาเป้าหมายของระบบต้นแบบว่าเน้นการแสดงสถานะและการแจ้งเตือนพื้นฐานก่อน
- การแจ้งเตือนต้องสอดคล้องกับสถานะเหตุการณ์และแสดงผลได้ชัดเจนใน UI
- ระบบต้องรองรับการใช้งานทั้งมือถือและเดสก์ท็อป
- ต้องมีการแสดงสถานะล่าสุดและประวัติการแจ้งเตือนอย่างเป็นระเบียบ
- ควรพิจารณาความปลอดภัยของข้อมูลและการเข้าถึงที่เหมาะสมในทุก API และหน้าเว็บ
- หลีกเลี่ยงการเพิ่มฟีเจอร์ที่ไม่เกี่ยวข้องกับขอบเขตของโปรเจกต์ต้นแบบโดยไม่จำเป็น
- หากเปลี่ยนแปลงโครงสร้างหรือสถาปัตยกรรม ควรอธิบายเหตุผลให้ชัดเจนก่อนดำเนินการ
- ก่อนส่งงานหรืออัปเดตหลัก ควรตรวจสอบว่า build และ lint ผ่านอยู่

## Implementation Priority
1. สร้างหน้าแสดงสถานะการตรวจจับควันและเปลวไฟ
2. สร้างกลไกแจ้งเตือนเมื่อเกิดเหตุการณ์ความเสี่ยง
3. แสดงประวัติการแจ้งเตือนย้อนหลังในรูปแบบที่อ่านง่าย
4. ปรับปรุง UI ให้ใช้งานง่ายและตอบสนองเร็วทั้งบนมือถือและเดสก์ท็อป
